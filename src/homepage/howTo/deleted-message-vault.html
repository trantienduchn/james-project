Deleted Message Vault

This document will help you understand what is Deleted Message Vault and what does it for. And finally, present to you, how to setup James Server with this feature.

What is Deleted Message Vault

Deleted Message Vault is a plugin in James, it allows you to:

  • store deleted messages from users
  • search for deleted messages by various criteria
  • delete a message by its id
  • purge all expired messages, expiration time is calculated from a given configured duration called retentionPeriod

If Deleted Message Vault is enabled, when users delete their mails, along with deleting mails in the mail system, James also stores those mails into Deleted Message Vault. And only administrators can interact with this component via WebAdmin rest APIs.

It is useful to keep users deleted messages, administrators can helps users to restore their deleted messages in case of delete by accident. But messages stored in Deleted Message Vault don't exist forever, Deleted Message Vault support to delete messages after given period had been configured (retentionPeriod property) by the administrators. It also supports to delete a single message by the message id.

Deleted Message Vault API allow to search messages by various criteria base on mail properties. At the moment, these are supported mail properties for searching:

  • deletion date(ISO-8601 Date String)
    • supports before or equals operator
    • supports after or equals operator
  • delivery date(ISO-8601 Date String)
    • supports before or equals operator
    • supports after or equals operator
  • recipients(List of string)
    • supports contains operator
  • sender(String)
    • supports equal matching operator
  • has attachment(Boolean)
    • supports equal matching operator
  • subject(String)
    • supports equal matching operator
    • supports equal ignore case matching operator
    • supports contains matching operator
    • supports contains ignore case matching operator (with US locale)

Make Deleted Message Vault available in James

To do this, you have to create a configuration file deletedMessageVault.properties, then put it into conf directory of James. There are two available properties you may want to configure:

  • urlPrefix: represent for the prefix of namespace Deleted Message Vault will use to store deleted messages. urlPrefix has pattern: mail repository protocol + "://" + mail repository prefix path. E.g: cassandra://var/deletedMessages/vault This property is optional, if you don't specify it, then default value will be used (James default mail repository protocol + "://var/deletedMessages/vault")
  • retentionPeriod: represent for the period deleted messages allowed to be stored in Deleted Message Vault. retentionPeriod is a duration string and will be parsed by org.apache.james.util.DurationParser. So it follows available patterns in DurationParser. E.g: "1 year", "10 months", "30 days",.... This property is optional, if you don't specify it, then default value will be used (1 year)

Make James uses Deleted Message Vault by configuring Pre Deletion Hooks

By default, although Deleted Message Vault has been configured, but, to make it really work, you still need to configure Pre Deletion Hooks to lets James use it. Before deleting a mail in James, PreDeletionHooks will be triggered to execute all hooks, if all hook executions success, then Jame will process to delete that mail. There is already a DeletedMessageVaultHook in James, it's job is storing deleted mails into Deleted Message Vault. Thus, you need to configure this hook in listener.xml configuration file.

Sample DeletedMessageVaultHook configuration:

<listeners> <listener> ... </listener> ... <preDeletionHook> <class>org.apache.james.vault.DeletedMessageVaultHook</class> </preDeletionHook> </listeners>

Start a James Server with Deleted Message Vault

We will take James cassandra product for the example

First, get template cassandra product configuration:


                        $ git clone https://github.com/apache/james-project
                        $ cp -rf james-project/dockerfiles/run/guice/cassandra/destination/conf conf
                        $ mv keystore conf/keystore

Second, modify deletedMessageVault.properties configuration file


                        urlPrefix=cassandra://var/deletedMessages/vault
                        retentionPeriod=1 year

Third, modify listener.xml to configure DeletedMessageVaultHook by adding preDeletionHook section at above paragraph

Fourth, We will create a local folder for holding data out of the container:

mkdir var

                        docker run
                        --name james_run \
                        --hostname HOSTNAME -p "25:25" -p 80:80 -p "110:110" -p "143:143" -p "465:465" -p "587:587" -p "993:993" \
                        --link cassandra:cassandra \
                        --link elasticsearch:elasticsearch \
                        --link tika:tika \
                        --volume "$PWD/conf:/root/conf/" \
                        --volume "$PWD/var:/root/var/" \
                        linagora/james-cassandra-guice:latest

WebAdmin Deleted Message Vault exporting

This part is a bit special to you, you are able to choose which exporting mechanism to be used. At the moment there are one available exporting mechanism

  • localFile
    • : This is a simple exporting mechanism while with a export request, it retrieve deleted messages from Deleted Message Vault, then store them as a zip file in local file system in James Server. Then sending a mail with the absolute path of exported file to the targeted mail address.

You can configure which kind of export mechanism to use in James by specifying blob.export.implementation in blobStore.properties configuration file. E.g: 


                        blob.export.implementation=localFile
